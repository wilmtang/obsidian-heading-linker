## A1 (P1, FIXED): Headings with `[` or `]` silently break find / rename / migrate

> **Status: Fixed.** The markdown-label groups in the find/rename scanner
> ([`main.ts:645`](main.ts:645)) and the migration scanner
> ([`main.ts:1049`](main.ts:1049)) are now escape-aware
> (`(?:\\.|[^\]\\\n])*`), so an escaped `\]` inside a label no longer ends the
> match early. Label comparison now runs through `unescapeMarkdownLinkText`
> before comparing to the old heading name. Regression tests cover copy →
> paste → rename for `A [x]`. Verified: rename now reports `count=1` and
> rewrites both the label and the destination.


`bug-audit.md` records this only as P2 ("labels can remain stale", with the
destination still updated). In practice it is worse: for a heading whose name
contains a bracket, the plugin's own generated link is **not matched at all**, so
find/rename/migration are complete no-ops and report success.

**Reproduction (captured from the real code paths):**

```
heading "A [x]"  ->  plugin copies:  [A \[x\]](<./A.md#A [x]>)
rename "A [x]" -> "Renamed":         [A \[x\]](<./A.md#A [x]>)   (count = 0, unchanged)
```

**Root cause.** When copying, the label is escaped via
[`escapeMarkdownLinkText`](main.ts:309) (`[` → `\[`, `]` → `\]`) — correct for
rendering. But the scanner's label group is not escape-aware:

- find/rename: [`main.ts:642`](main.ts:642) — `\[([^\]\n]*)\]\(...`
- migration: [`main.ts:1046`](main.ts:1046) — `(\[[^\]\n]*\]\()...`

`[^\]\n]*` stops at the first literal `]`, including the escaped `\]` the plugin
itself wrote. The link start position therefore fails to match and no rewrite is
attempted. Separately, even if the link matched, the label comparison
[`label === target.oldName`](main.ts:649) compares the escaped label against the
raw heading text and would never be true.

**Impact.** Headings like `[Draft] Title` or `TODO [P1]` produce links that the
plugin can neither find nor refactor, and rename reports "No links to update"
(or an undercount) with no warning — a silent correctness failure of the
headline feature.

**Suggested fix.** Make the label group escape-aware, e.g.
`\[((?:\\.|[^\]\\\n])*)\]`, and unescape the captured label before comparing to
`oldName` / re-escape with `escapeMarkdownLinkText(newName)` when rewriting.
Apply the same regex in the migration scanner. Add regressions for `A [x]` and
`[Draft] Note`.

## A2 (P2, FIXED): Generated angle-bracket destinations escape `>` but not `<`

> **Status: Fixed.** [`escapeMarkdownLinkDestinationContent`](main.ts:301) now
> escapes both `<` and `>` (`/[<>]/g`), and
> [`unescapeMarkdownLinkDestinationContent`](main.ts:305) reverses both, keeping
> the pair symmetric. A heading `A < B > C` now copies as
> `[A < B > C](<./Target.md#A \< B \> C>)`. Covered by a new generation test and
> a `<` round-trip rewrite test.

The `>` escaping landed in `25caca8`, but its sibling `<` was not handled.

**Reproduction:**

```
heading "A < B"  ->  [A < B](<./Target.md#A < B>)     (unescaped '<' inside <...>)
heading "A > B"  ->  [A > B](<./Target.md#A \> B>)    ('>' escaped, for contrast)
```

**Root cause.** [`escapeMarkdownLinkDestinationContent`](main.ts:301) replaces
only `>`; [`unescapeMarkdownLinkDestinationContent`](main.ts:305) only `\>`.
[`formatMarkdownLinkDestination`](main.ts:316) then wraps in `<...>`. A CommonMark
angle-bracket destination may not contain an unescaped `<` or `>`, so a heading
or filename containing `<` yields a destination that is likely mis-parsed by
Obsidian's link resolver/renderer.

**Impact.** Headings/filenames with `<` (e.g. `Vector<T>`, `a<b`) generate links
that may not resolve or render. Lower frequency than `>` but the same class of
bug the `>` fix was meant to close. (The internal scanner still matches these,
since its destination group allows `<`; the breakage is on the Obsidian side.)

**Suggested fix.** Escape `<` alongside `>` in
`escapeMarkdownLinkDestinationContent`, and unescape `\<` in
`unescapeMarkdownLinkDestinationContent`, keeping the two symmetric. Extend the
`A > B` regression to cover `A < B`.

## A3 (P2, FIXED): No error handling around the clipboard write; file already mutated

> **Status: Fixed.** [`copyHeadingLink`](main.ts:241) now defers the in-document
> edit until after a successful clipboard write. The write is wrapped in
> `try/catch`; on failure it logs, shows a "Failed to copy heading link to
> clipboard." notice, and returns without mutating the document. The success
> notice only fires after the (possibly deferred) edit is applied. Regression
> test asserts the editor is untouched when the write rejects.


[`copyHeadingLink`](main.ts:241) inserts a stable target into the live document
for duplicate headings ([`main.ts:265`](main.ts:265)) and only then calls
`await navigator.clipboard.writeText(...)` ([`main.ts:277`](main.ts:277)). The
success `Notice` is on the line after.

The command/menu call sites invoke it as `void this.copyHeadingLink(...)`
([`main.ts:142`](main.ts:142), [`main.ts:172`](main.ts:172)). If the clipboard
write rejects (no focus, denied permission, non-secure context), then:

1. The heading line has **already** been modified (a `^id` was inserted).
2. No success or failure `Notice` is shown.
3. The rejected promise is swallowed by `void`.

**Impact.** The user's note is edited but nothing reaches the clipboard, with no
feedback — they may not realize a stray block id was added or that the copy
failed.

**Suggested fix.** Wrap the write in `try/catch`, show an error `Notice` on
failure, and consider building the link string before mutating the document so a
failed copy can avoid (or clearly report) the in-document change.

## A4 (P2, FIXED): Markdown label goes stale on rename (backslash case)

> **Status: Fixed together with A1.** Label comparison now unescapes the raw
> label via `unescapeMarkdownLinkText` before comparing to `oldName`, so an
> escaped display label that renders as the old heading is updated and
> re-escaped. Regression test added.

Carried from `bug-audit.md` P2. For a heading name that escapes to a label
**without** a `]` (so the link still matched), the destination updated but the
label did not:

```
heading "A \ B"  ->  [A \\ B](<./A.md#A \ B>)
before fix, rename -> "Renamed":  [A \\ B](<./A.md#Renamed>)   (label stale)
after fix,  rename -> "Renamed":  [Renamed](<./A.md#Renamed>)  (label updated)
```

Root cause was A1's comparison half: [`main.ts:652`](main.ts:652) compared the
raw (escaped) label to `oldName`. This restores the README promise that matching
display aliases are updated.

## A5 (P2, CONFIRMED): Wikilink rename to names with `|`, `]]`, or newline corrupts links

Carried from `bug-audit.md` P2. The rename modal accepts any trimmed string and
wikilink rewriting injects it raw:

- subpath: [`main.ts:625`](main.ts:625) — `replaceSubpath(linkTarget, '#' + target.newName)`
- alias: [`main.ts:627`](main.ts:627)

**Reproduction** — renaming heading `Intro` to `B | C]] x`:

```
[[A#Intro]]  ->  [[A#B | C]] x]]
```

The embedded `]]` closes the wikilink early and `|` introduces a spurious alias.

**Suggested fix.** Reject or sanitize new heading names containing wikilink
syntax (`|`, `]]`, `#`, newline) in the rename modal, or escape them when
constructing wikilinks. Markdown links are safer here only because labels are
escaped.

## A6 (P3, NEW): No `versions.json`

The repo has no `versions.json`, and the release workflow ships only `main.js`
and `manifest.json`. Obsidian uses `versions.json` to map plugin versions to a
`minAppVersion` so older app installs resolve a compatible plugin version; its
absence is a packaging gap for a community plugin (manifest already declares
`minAppVersion: 1.5.7`).

**Suggested fix.** Add `versions.json` (e.g. `{ "1.0.7": "1.5.7" }`), include it
in the `softprops/action-gh-release` `files:` list, and consider extending
`scripts/check-versions.mjs` to assert the current `manifest.version` key exists
in it.

## A7 (P3, NEW): Pervasive inline `style` attributes

The modals and suggestion renderer set presentation through inline
`attr: { style: '...' }` strings throughout (e.g.
[`main.ts:1371`](main.ts:1371), [`main.ts:1440`](main.ts:1440),
[`main.ts:1535`](main.ts:1535)). ESLint passes today, but the Obsidian
community-plugin guidelines discourage JS/inline styling in favor of CSS classes
in `styles.css`. This is maintainability/review-readiness debt, not a runtime
bug. (Dynamic DOM text already uses `createEl`/`createTextNode`, so there is no
HTML-injection risk.)

**Suggested fix.** Move static styles into a bundled `styles.css` and apply
class names; keep only genuinely dynamic values inline.

## A8 (P3, NEW): Trailing-newline disagreement between the two line splitters

[`splitLines`](main.ts:825) (`String.split`) yields a trailing empty element for
content ending in a newline, while [`splitLinesWithEndings`](main.ts:483) does
not (it folds the final newline into the previous line's `ending`). The
migration plan indexes lines with `splitLines` ([`main.ts:1112`](main.ts:1112))
but applies edits against `splitLinesWithEndings`
([`main.ts:1594`](main.ts:1594)).

This is currently **harmless** because the only differing element is a trailing
empty line that never carries a planned change, so real change indices still
align. It is flagged as latent fragility: any future change that records edits
on the last line, or that relies on equal line counts, could mis-target. A
shared splitter (or a comment documenting the invariant) would remove the
footgun.
